# Модель сущности профиля

Документ фиксирует текущую структуру данных прототипа, чтобы создание профиля, таблица профилей и карточка профиля работали с одной сущностью.

## Profile

| Поле | Источник | Назначение |
| --- | --- | --- |
| `id` | система | Внутренний идентификатор профиля. |
| `code` | система | Код профиля в формате `PROF-001`. |
| `name` | создание/карточка | Наименование профиля должности. |
| `department` | создание/карточка | Место профиля в структуре. |
| `band` | система | Бэнд должности, пока задаётся значением по умолчанию. |
| `hasBinding` | система | Признак привязки к позиции. |
| `classifier` | создание/карточка | Значение классификатора. |
| `isTypicalPosition` | система | Признак типовой должности. |
| `updatedAt` | система | Дата последнего изменения. |
| `status` | система/карточка | Состояние профиля: черновик, на оценке, архив и другие. |
| `okzCode` | создание/карточка | Код ОКЗ. |
| `categories` | система/карточка | Категории профиля. |
| `isTypicalProfile` | система | Признак типового профиля. |
| `leaderName` | система/карточка | Руководитель профиля. |
| `leaderPosition` | система/карточка | Должность руководителя. |
| `goals` | создание/карточка | Цели, задачи и функции профиля. |
| `competencies` | создание | Компетенции и квалификационные требования второго этапа. |
| `competenciesAreUserDefined` | создание/система | Признак, что компетенции пришли из пользовательского сценария создания или редактирования. Если `true`, пустые разделы компетенций не заменяются дефолтными mock-значениями. |

## Goals

| Поле | Назначение |
| --- | --- |
| `name` | Наименование цели. |
| `role` | Роль участия профиля в цели. |
| `tasks` | Список задач внутри цели. |

## Tasks

| Поле | Назначение |
| --- | --- |
| `name` | Наименование задачи. |
| `participation` | Доля участия задачи в цели. |
| `role` | Роль профиля в задаче. |
| `functions` | Список функций внутри задачи. |

## Связь справочников целей и задач

В интерфейсе создания профиля список значений в поле `Наименование задачи` зависит от выбранного наименования цели в той же карточке цели.

Если для выбранной цели задана связка с задачами, выпадающий список задачи показывает только связанные задачи без дополнительных групп, заголовков или пояснений. Если цель не выбрана или для нее нет связки, используется общий список задач.

Пользовательская задача, созданная из дропдауна задачи при выбранной цели, сохраняет привязку к этой цели и повторно отображается в списке задач этой цели.

## Связь справочников задач и функций

В интерфейсе создания профиля список значений в поле `Наименование функции` зависит от выбранного наименования задачи в той же строке задачи.

Если для выбранной задачи задана связка с функциями, выпадающий список функции показывает только связанные функции без дополнительных групп, заголовков или пояснений. Если задача не выбрана или для нее нет связки, используется общий список функций.

Пользовательская функция, созданная из дропдауна функции при выбранной задаче, сохраняет привязку к этой задаче и повторно отображается в списке функций этой задачи.

Уже выбранные функции не очищаются автоматически при смене задачи. Возможное несоответствие выбранной функции новой задаче должно обрабатываться через анализ профиля, а не через принудительный сброс значения.

## Автозаполнение компетенций

Если у базового или ранее сохраненного системного профиля отсутствуют выбранные значения в одном из разделов компетенций, модель профиля автоматически подставляет базовый набор значений для этого раздела. Это гарантирует, что карточки профилей базового списка на вкладке «Ключевые компетенции» не показывают пустые аккордеоны.

Для пользовательского сценария создания нового профиля пустые массивы компетенций сохраняются как пустые значения. Это нужно, чтобы AI-анализ видел незаполненный второй этап и мог предложить точечное дозаполнение, а не принимал дефолтные mock-компетенции за выбор пользователя.

Технический маркер такого сценария — `competenciesAreUserDefined: true`.
